# Технический обзор решения Baby Tracker

Baby Tracker построен как клиент-серверная система с единым источником данных в **Supabase (PostgreSQL + Auth)**. Клиентская часть — это **React 18 + Vite** приложение, которое работает в браузере, внутри Telegram WebApp и как Android-приложение через **Capacitor**. На фронтенде есть слой сервисов (`supabaseService`, `notificationService`, `cacheService`), который отвечает за чтение/запись данных, кеширование и адаптацию поведения под платформу (web/telegram/android). Это позволяет использовать один UI-код и один домен данных на всех каналах. 

В центре архитектуры находится БД Supabase. Основная операционная таблица — `activities` (журнал событий ребёнка), а также пользовательские сущности и связь Telegram-аккаунтов. Для аналитики применён гибрид: часть метрик считается на клиенте для быстрых экранов, а тяжёлые агрегаты считаются в БД и хранятся в `daily_activity_stats` и `activity_interval_stats`. SQL-функции и триггеры на `INSERT/UPDATE/DELETE` в `activities` автоматически пересчитывают только затронутые даты и типы активностей, поэтому статистика остаётся консистентной и масштабируется при росте данных.

Сервисная интеграция организована отдельными Node.js компонентами. **Telegram-бот** (на `node-telegram-bot-api`) обслуживает чат-интерфейс и быстрые сценарии ввода активностей, используя те же таблицы Supabase, что и веб/мобайл клиент. **Notification server** (Express + `node-cron` + Firebase Admin) обрабатывает пуш-уведомления и расписания: Android-клиент регистрирует FCM-токен и сохраняет его в Supabase, сервер по расписанию выбирает релевантные напоминания из БД и отправляет push через FCM; для Telegram уведомления отправляются через бота. В итоге все каналы (web, Android, Telegram) работают поверх одной БД и синхронизируются через единый backend-as-a-service слой.

Ключевое различие Android и Telegram-режимов — в окружении выполнения и канале уведомлений. В **Android** приложение запускается как нативный контейнер Capacitor: доступны плагины устройства (Push/Local Notifications, Preferences, Network), есть регистрация FCM-токена и получение push даже вне Telegram. В **Telegram WebApp** приложение работает внутри встроенного браузера Telegram, не использует Capacitor API и опирается на Telegram Bot API для напоминаний, быстрых сценариев и открытия приложения из чата.

Также различается поток авторизации и пользовательский путь: в Android/web основной вход — через email/password в Supabase Auth, после чего пользователь работает в полноэкранном приложении; в Telegram пользователь может начать взаимодействие с ботом, пройти привязку аккаунта и запускать базовые действия прямо из чата. При этом данные и в Android, и в Telegram сохраняются в одни и те же таблицы Supabase, поэтому история активностей и статистика остаются общими и консистентными.


По инфраструктуре деплоя: **фронтенд развернут на Vercel**, а **notification-server и Telegram-бот развернуты на Render**. Проект использует бесплатные тарифы на этих платформах, поэтому есть эксплуатационные ограничения: возможны «засыпание» инстансов и холодный старт, ограниченные ресурсы CPU/RAM и квоты, а также менее предсказуемая латентность фоновых задач (например, проверка/доставка уведомлений). Это важно учитывать при ожиданиях по SLA и при планировании масштабирования на платные планы.

## Структура проекта: ключевые папки и файлы

Ниже — «карта» проекта, чтобы быстро понять где находится основной функционал:

- `src/main.jsx` — точка входа фронтенда, инициализация React-приложения.
- `src/App.jsx` — корневой компонент приложения и маршрутизация основных экранов.
- `src/components/` — UI-экраны и виджеты (главный экран, профиль, уведомления, статистика, формы активностей).
- `src/services/supabaseService.js` — обёртка над запросами к Supabase с кешированием и инвалидацией кеша.
- `src/services/notificationService.js` — инициализация и обработка уведомлений на разных платформах (особенно Android/FCM).
- `src/services/cacheService.js` — локальный кеш для снижения числа повторных запросов и ускорения UI.
- `src/utils/supabase.js` — создание Supabase-клиента, auth-helper’ы (email auth, привязка Telegram и др.).
- `src/utils/platform.js` — определение среды выполнения (web / telegram / android) и платформенные проверки.
- `notification-server/notification-server.js` — сервер уведомлений (cron-проверка расписаний, отправка push/Telegram-уведомлений).
- `bot/bot.js` — Telegram-бот (команды, меню, быстрый ввод активностей, привязка аккаунта).
- `scripts_statistics_aggregates.sql` — SQL-структуры и функции для агрегированной статистики + триггеры пересчёта.
- `scripts_safe_migration_user_telegram_mapping.sql` — безопасная миграция/нормализация связки Telegram ↔ пользователь.
- `android/` и `capacitor.config.json` — Android-обёртка Capacitor и нативная конфигурация сборки.
- `package.json` (корень) — скрипты сборки фронтенда (`dev`, `build`, `build:android`) и зависимости web-клиента.
- `notification-server/package.json`, `bot/package.json` — зависимости и команды запуска серверных компонентов.

Такая структура отражает разделение ответственности: UI и клиентская логика в `src/`, серверные интеграции в `bot/` и `notification-server/`, а изменения схемы и вычислений БД — в SQL-скриптах.
